# sav_ds — Component Stress-Test Report

Generated by the dedicated stress harness `test/stress_test.dart`
(run: `flutter test test/stress_test.dart`). The harness is **diagnostic**, not a
gate — it drains layout exceptions with `tester.takeException()` so it records
overflows/errors instead of failing, and prints one `STRESS|…` line per scenario.

Three axes:
- **VOLUME** — 200 instances built eagerly in a `Column` (build cost + controller churn).
- **EXTREME** — pathological single inputs (1000-char labels, huge amounts, 30 tabs…).
- **CHURN** — 50× mount/unmount (dispose / leak correctness).

> ⚠️ Timings are **widget-build times in the headless test binding** (no GPU/raster).
> Treat them as *relative* indicators between components, not real frame times.

## Root-cause analysis & possible fixes

### Issue 1 — `AppButton` overflows with a long label
- **Symptom:** 1000-char label → `RenderFlex overflowed by 13704px`.
- **Cause:** the label is rendered by `MorphingText` ([morphing_text.dart](sav_ds/lib/components/morphing_text.dart)),
  which lays every character out as a separate `Text` inside a `Row(mainAxisSize: MainAxisSize.min)`
  — an intrinsically **unbounded width** with no clipping. `AppButton`'s content `Row`
  ([app_button.dart](sav_ds/lib/components/app_button.dart), the `content`/`rowWidget` block) places
  that `MorphingText` **without a `Flexible`/`Expanded`**, so nothing constrains it to the button width.
- **Possible fixes:**
  1. Wrap `MorphingText` in `Flexible` inside the button row, and have `MorphingText` cap to one line
     with `TextOverflow.ellipsis` (it would need to fall back to a plain `Text` when the string can't
     fit, since per-glyph animation can't ellipsis natively).
  2. Simpler/robust: clip the row (`ClipRect`) and/or `FittedBox(fit: BoxFit.scaleDown)` around the
     label so long text shrinks instead of overflowing.
  - **Recommended:** `Flexible` + ellipsis (1); buttons should truncate, not shrink.

### Issue 2 — `SavChip` overflows with a long label
- **Symptom:** 1000-char label → `RenderFlex overflowed by 11692px`.
- **Cause:** [sav_chip.dart](sav_ds/lib/components/sav_chip.dart) renders
  `Row(mainAxisSize: MainAxisSize.min, children: [ …, Text(widget.label) ])` — the label `Text` has
  **no `Flexible` wrapper and no `overflow`**, so the chip grows to the full text width and overflows
  its parent.
- **Possible fixes:**
  1. Wrap the label in `Flexible(child: Text(label, overflow: TextOverflow.ellipsis, maxLines: 1))`.
  2. Add a `maxWidth` constraint to the chip (chips are meant to be compact).
  - **Recommended:** (1) — one-line ellipsis is the conventional chip behavior.

### Issue 3 — `SelectableRow` overflows **vertically** with a long label
- **Symptom:** 1000-char label → `RenderFlex overflowed by 545px on the bottom`.
- **Cause:** the row is a `Container(height: 56, …)` (fixed) in
  [selectable_row.dart](sav_ds/lib/components/selectable_row.dart), but the label lives in an
  `Expanded → Column → Text(widget.label)` with **no `maxLines`/`overflow`**. `Expanded` bounds the
  width so the long label **wraps to many lines**, the `Column` grows past 56px, and the fixed height
  clips → bottom overflow.
- **Possible fixes:**
  1. Give the label `maxLines: 1/2` + `TextOverflow.ellipsis` (same for `secondary`).
  2. Replace the hard `height: 56` with `ConstrainedBox(minHeight: 56)` so the row grows to fit
     multi-line content instead of clipping.
  - **Recommended:** (1) for list density; optionally (2) if multi-line labels are desired.

### Issue 4 — `SegmentedControl` throws with many items
- **Symptom:** 30 items → 30 layout exceptions.
- **Cause:** in the pill/full-width path ([segmented_control.dart](sav_ds/lib/components/segmented_control.dart))
  each segment is wrapped in `Expanded(child: button)`, so 30 segments each get `width/30` (~13px),
  but each segment's `Text` content needs far more and **overflows inside every segment** → one
  exception per segment. The component assumes a small, screen-fitting set. (An underline variant has
  a `SingleChildScrollView`, but the pill path does not.)
- **Possible fixes:**
  1. When `isFullWidth` and items don't fit, fall back to the scrollable layout (`SingleChildScrollView`
     + intrinsic-width segments) instead of `Expanded`.
  2. Add `overflow: ellipsis`/`Flexible` to each segment's label so tight segments truncate rather
     than throw.
  3. Document/enforce a sensible max item count (segmented controls are designed for 2–5 options).
  - **Recommended:** (2) to stop the exceptions + (3) as guidance; (1) if many items is a real use case.

### Issue 5 — `AmountInput` clips huge values (doesn't shrink)
- **Symptom:** `999999999999999.99` → no error but the number is **hard-clipped**.
- **Cause:** the amount sits in a fixed `SizedBox(width: 160)` with `clipBehavior: Clip.hardEdge` at a
  fixed 48px font, inside a `SingleChildScrollView(NeverScrollableScrollPhysics)`
  ([amount_input.dart](sav_ds/lib/components/amount_input.dart), `_buildInputArea`). Long values
  exceed 160px and are clipped — they neither scroll nor scale.
- **Possible fix:** measure the text and **auto-shrink the font to fit** (apply the same scaled style
  to the visible text *and* the transparent `TextField` so the caret stays aligned). Already scoped as
  a separate task.

### Issue 6 — Controller-heavy components are ~10× costlier in volume
- **Symptom:** OTPInput 833 ms / InputField 756 ms / AmountInput 652 ms / AppButton 627 ms for 200,
  vs SavChip/SavBadge ~57 ms.
- **Cause:** each instance spins up `AnimationController`(s) + `TextEditingController`/`FocusNode` +
  listeners (and `AppButton` adds `NoiseLayer` + `RepaintBoundary`). That's correct for interactive
  use but expensive when **eagerly** building hundreds.
- **Possible fixes:**
  1. Document that long lists must use `ListView.builder` (lazy) — only visible instances build.
  2. Offer a lightweight static/preview `AppButton` (no `AnimationController`, no noise) for dense,
     non-interactive grids.
  - Not a bug; a usage + optional-variant recommendation.

## Priority
1. **Issues 1–3 (label overflow guards)** — easy, high-value robustness; same fix pattern
   (`Flexible` + `maxLines`/`ellipsis`).
2. **Issue 4 (SegmentedControl)** — stop the exceptions with per-segment ellipsis; scroll fallback if needed.
3. **Issue 5 (AmountInput auto-shrink)** — already scoped.
4. **Issue 6** — docs + optional static variant.

## Re-run / extend
- `flutter test test/stress_test.dart` regenerates the `STRESS|…` lines.
- Tune `volumeN` / `longLabel` / `hugeAmount` at the top of the harness.
- For **real GPU/raster** perf (true frame times, jank), add `integration_test --profile` or a
  "Stress" screen in `example/` driven on a device — the build numbers here are only relative.
